Итоговое решение это гибрид трёх методов поиска (BM25, LSA и нейросетевые
эмбеддинги), результаты которых объединяются по рангам. На размеченной выборке
`calibration.f` оно даёт MAP@10 около 0.44 по честной кросс-валидации.

## Что где лежит
 
```
src/
  metrics.py          подсчёт метрики MAP@10 и вспомогательный AP@10
  test_metrics.py     10 проверок метрики на примерах, посчитанных вручную
  text_processing.py  очистка HTML и токенизация текста
  retrieval.py        сигналы поиска (BM25, LSA, эмбеддинги) и их объединение
  solution.py         основной скрипт: валидация на calibration и генерация answer.csv
  eda.ipynb           разведочный анализ данных
  lsa.ipynb           эксперименты с LSA
  rrf.ipynb           эксперименты с объединением сигналов через RRF
candidate_public/candidate_data/   исходные данные (articles.f, calibration.f, test.f)
answer.csv            готовый ответ для test.f
pyproject.toml        зависимости (проект собран через uv)
```
 
Файл `main.py` это заглушка, созданная при инициализации проекта, в работе не
используется. Точка входа это `src/solution.py`.
 
## Установка и запуск
 
Проект использует uv. Установка зависимостей и запуск:
 
```
uv sync
uv run python src/solution.py            # BM25 + LSA, работает без интернета
uv run python src/solution.py --use-emb  # добавить нейроэмбеддинги e5
```
 
Если uv не используется, подойдёт обычный pip по списку из `pyproject.toml`.
Скрипт сам находит папку с данными, но её можно указать явно через `--data ПУТЬ`.
 
Для варианта `--use-emb` при первом запуске скачивается модель
`intfloat/multilingual-e5-base` с Hugging Face (примерно 1 ГБ), дальше всё
считается локально на CPU. Внешние сервисы для получения ответов не
используются.
 
Решение детерминировано (зафиксирован seed 42), поэтому повторный запуск даёт
тот же самый `answer.csv`.
 
## Обработка HTML статей
 
Тексты статей хранятся в HTML с разметкой, таблицами, вкладками и спойлерами.
Я привожу их к плоскому тексту функцией `clean_html` в `text_processing.py`:
разбираю HTML через BeautifulSoup с парсером lxml, удаляю содержимое тегов
`script` и `style`, а из остального забираю только текст. Куски из соседних
тегов склеиваю через пробел, чтобы слова не слипались, и схлопываю лишние
пробелы и переносы строк. Текст внутри таблиц, вкладок и спойлеров при этом
сохраняется, теряется только разметка.
 
Проверка на calibration подтвердила, что чистка помогает: BM25 по сырому HTML
давал MAP@10 около 0.28, а по очищенному тексту около 0.30.
 
Токенизация (функция `tokenize`) переводит текст в нижний регистр, заменяет ё
на е, вырезает слова регулярным выражением (заглушки вида `<MONEY>` и `<ID>`
при этом отсекаются), убирает стоп-слова и при необходимости применяет стемминг.
Стоп-слова беру из готового пакета `stop-words`, а не из своего списка, чтобы не
было вопросов о том, не подгонял ли я их под данные.
 
## Признаки, модели и алгоритмы
 
Решение построено как поиск в два логических шага: сначала отбор релевантных
статей несколькими независимыми методами, затем объединение их результатов.
 
Для документа я склеиваю заголовок и очищенное тело статьи, причём заголовок
повторяю три раза, чтобы придать ему больший вес.
 
Используются три сигнала поиска, все они лежат в `retrieval.py`:
 
1. BM25 (библиотека rank_bm25). Классический лексический поиск по совпадению
   слов с учётом их редкости, насыщения частоты и длины документа. Работает без
   стемминга, потому что на этих данных стемминг снижал качество (см. ниже).
   Параметры k1=1.2, b=0.4.
2. LSA (латентно-семантический анализ). Строю матрицу TF-IDF, затем понижаю
   размерность через усечённое сингулярное разложение (TruncatedSVD, 100
   компонент) и сравниваю запрос со статьями по косинусной близости в этом
   пространстве. LSA ловит связи между словами по совместной встречаемости и
   частично закрывает разрыв словаря, когда пользователь и статья пишут об одном
   разными словами.
3. Нейросетевые эмбеддинги (sentence-transformers, модель
   `intfloat/multilingual-e5-base`, менее 1B параметров). Кодирую запрос и статьи
   в векторы и сравниваю по косинусной близости. Модель заранее обучена на
   большом объёме текста и понимает близость смыслов лучше LSA. Так как статьи
   бывают очень длинными, а модель читает ограниченное число токенов, я режу тело
   статьи на куски примерно по 180 слов с перекрытием, кодирую каждый кусок с
   заголовком, а близость статьи к запросу беру как максимум по её кускам. Без
   такого разбиения длинные статьи обрезались до начала и качество падало.
Результаты трёх сигналов объединяю через Reciprocal Rank Fusion (RRF). RRF
работает не с сырыми оценками, у которых разный масштаб, а с позициями: каждый
сигнал даёт документу вклад 1 / (k + позиция), вклады складываются с весами, и
по сумме строится итоговый список. Веса подбираются автоматически на
calibration по кросс-валидации.
 
Смысл объединения в том, что методы ошибаются в разных местах. BM25 хорошо
находит точные совпадения слов, но не понимает синонимы. Эмбеддинги и LSA
понимают смысл, но проигрывают на точных терминах и номерах. По отдельности
каждый метод даёт MAP@10 около 0.28 - 0.31, а вместе около 0.44.
 
## Проверка качества на calibration.f
 
Метрику MAP@10 я реализовал сам в `metrics.py`, строго по формуле из задания, и
покрыл её десятью проверками в `test_metrics.py` на примерах, где ответ можно
посчитать вручную. Это нужно, чтобы локальная оценка совпадала с оценкой
платформы и чтобы можно было проверять идеи у себя, не тратя попытки отправки.
 
Файл `calibration.f` с известными правильными ответами служит проверочной
выборкой. Чтобы не переобучиться под неё при подборе весов объединения, я
использую кросс-валидацию по запросам (5 фолдов): веса выбираются на обучающих
фолдах и оцениваются на отложенных.
 
Основные числа MAP@10 на calibration:
 
- одинаковый список самых частых статей всем запросам (тривиальный ориентир): 0.32
- BM25 по очищенному тексту: 0.31
- LSA отдельно: 0.28
- эмбеддинги отдельно: 0.29
- BM25 + LSA, объединение по кросс-валидации: 0.36
- BM25 + LSA + эмбеддинги, объединение по кросс-валидации: 0.44
Тривиальный ориентир здесь неожиданно сильный из-за перекоса в данных: несколько
статей являются правильным ответом для очень многих запросов. Я специально не
опираюсь на популярность статей как на основной сигнал, потому что на тестовой
выборке распределение может отличаться, а в реальном продукте новую статью без
истории тоже нужно находить по содержанию.
 
## Анализ ошибок
 
Я разобрал запросы, на которых базовый BM25 ошибался сильнее всего. Примерно у
трети запросов в топ-10 не попадала ни одна правильная статья. При этом почти во
всех таких случаях правильная статья все же находилась поиском, просто стояла
ниже топ-10, в среднем около двадцатой позиции. То есть BM25 неплохо отбирает
кандидатов, но плохо расставляет их по порядку. Только у малой доли промахов
(около 4 процентов) у запроса вообще не было ни одного общего слова с правильной
статьёй.
 
Ошибки я разделил на несколько типов:
 
- Разрыв словаря. Пользователь и статья пишут об одном разными словами.
  Например, вопрос про то, что будет, если не забрать заказ, а правильная статья
  называется про отмену заказа. Общих слов почти нет.
- Слова-дистракторы. В запросе есть яркое конкретное слово, которое уводит поиск
  в сторону. Например, вопрос про отправку бампера автомобиля, где слова про авто
  тянут выдачу к статьям про автомобили вместо статьи о том, что вообще можно
  отправлять.
- Статьи-близнецы. В базе есть несколько похожих статей на одну тему, и
  лексический поиск выбирает не ту из пары.
- Ложное совпадение слова. Явное совпадение по слову оказывается ошибочным.
  Например, вопрос про бонусы, где по слову подходит статья про бонусы, а
  правильный ответ по разметке лежит в статьях про оплату.
Общая причина у всех типов одна: лексический поиск работает по буквам, а
релевантность здесь определяется смыслом. Поэтому я и добавил к BM25 два
смысловых сигнала. LSA и эмбеддинги поднимают правильные статьи, которые BM25
держал около двадцатой позиции, в топ-10. На тех же примерах, где BM25 ставил
нужную статью на 15 или 72 место, эмбеддинги ставили её в первую десятку, и
объединение их выводило наверх.
 
## Что пробовал и отбросил
 
- Стемминг (обрезание окончаний) звучит полезно, но на calibration он снижал
  MAP@10 примерно с 0.30 до 0.25, потому что русский стеммер слишком грубо
  склеивал разные слова. Поэтому в BM25 стемминг выключен.
- Кросс-энкодер как вторая ступень переранжирования пробовал на топ-50
  кандидатах. Многоязычная модель не подошла под домен и опустила MAP@10 с 0.44
  до 0.31, поэтому этот шаг я не включал в итоговое решение.
## Использованные open-source библиотеки и модели
 
- rank_bm25 для BM25
- scikit-learn (TfidfVectorizer, TruncatedSVD) для LSA
- sentence-transformers и модель intfloat/multilingual-e5-base для эмбеддингов
- beautifulsoup4 и lxml для очистки HTML
- nltk (SnowballStemmer) и пакет stop-words для токенизации
- pandas, numpy, pyarrow для работы с данными
Все модели разворачиваются локально, запросы никуда не отправляются.
